# 网域层

网域层是位于界面层和数据层之间的可选层。可以降低UI层的复杂度，同时减少代码重复。

## 命名惯例

动词原形 + 名词/内容（可选）+ 用例。

例如：`FormatDateUseCase`、`LogOutUserUseCase`、`GetLatestNewsWithAuthorsUseCase` 或 `MakeLoginRequestUseCase`。



## 在 Kotlin 中调用用例

```kotlin
class FormatDateUseCase(userRepository: UserRepository) {

    private val formatter = SimpleDateFormat(
        userRepository.getPreferredDateFormat(),
        userRepository.getPreferredLocale()
    )

    operator fun invoke(date: Date): String {
        return formatter.format(date)
    }
}
```

使用

```kotlin
class MyViewModel(formatDateUseCase: FormatDateUseCase) : ViewModel() {
    init {
        val today = Calendar.getInstance()
        val todaysDate = formatDateUseCase(today)
        /* ... */
    }
}
```

用例没有自己的生命周期，而是受限于使用它们的类。您可以从界面层中的类、服务或 `Application` 类本身调用用例。由于用例不应包含可变数据，因此**您每次将用例类作为依赖项传递时，都应该创建一个新实例**。



## 线程处理

```kotlin
class MyUseCase(
    private val defaultDispatcher: CoroutineDispatcher = Dispatchers.Default
) {

    suspend operator fun invoke(...) = withContext(defaultDispatcher) {
        // Long-running blocking operations happen on a background thread.
    }
}
```



## 可重复使用的简单业务逻辑

在某些情况下，用例中可能存在的逻辑可以包含在 `Util` 类中的静态方法内。不建议采用`Util` 这种方式，因为 `Util` 类通常很难找到，而且其功能也很难发现。此外，用例还可以共享通用功能（例如基类中的线程处理和错误处理），这对规模较大的大型团队很有助益。

案例分析之合并两个仓库的数据：

```kotlin
/**
 * This use case fetches the latest news and the associated author.
 */
class GetLatestNewsWithAuthorsUseCase(
  private val newsRepository: NewsRepository,
  private val authorsRepository: AuthorsRepository,
  private val defaultDispatcher: CoroutineDispatcher = Dispatchers.Default
) {
    suspend operator fun invoke(): List<ArticleWithAuthor> =
        withContext(defaultDispatcher) {
            val news = newsRepository.fetchLatestNews()
            val result: MutableList<ArticleWithAuthor> = mutableListOf()
            // This is not parallelized, the use case is linearly slow.
            for (article in news) {
                // The repository exposes suspend functions
                val author = authorsRepository.getAuthor(article.authorId)
                result.add(ArticleWithAuthor(article, author))
            }
            result
        }
}
```

- 该用例使用默认调度程序将工作移到后台线程。

这个过程是遍历news列表的每一个article元素，然后对于每一个article元素，通过作者id再次查询作者的详细信息。

这个过程涉及对两个仓库的分别查询，属于是数据库查询的典型N+1问题。1次是获得所有articles，然后每个article对应一次后续查询。

如果能够自主控制后端API，最好是后端主动提供一个API接口，不要让前端多次查询来组合数据。

例如后端使用join表来一次查询出所需的数据。

```sql
SELECT au.id, au.name, ar.title, ...
FROM Author au
LEFT JOIN Article ar ON au.id = ar.author_id;
```

结论：可以借助 [Room 库](https://developer.android.com/training/data-storage/room?hl=zh-cn)，您可以查询数据库中不同实体之间的[关系](https://developer.android.com/training/data-storage/room/relationships?hl=zh-cn)。如果数据库是[可信来源](https://developer.android.com/jetpack/guide/data-layer?hl=zh-cn#source-of-truth)，您可以创建一个查询，让系统为您执行所有工作。在这种情况下，最好创建仓库类（例如 `NewsWithAuthorsRepository`），而不是用例。
