## 移动端跨平台技术

- **WebView 容器时代 (性能较差)**：核心是 `WebView`。基于此的框架如 **PhoneGap/Cordova/Ionic**，本质是把网页应用“包”成一个 App，虽然开发成本低，但受限于 WebView 本身的渲染效率和设备能力，体验较差。
- ** 原生桥接 (Hybrid) 时代 **：代表是 **React Native** 和 **Weex**。用 JavaScript 调用原生组件进行渲染，比 WebView 流畅。跨语言桥接在复杂场景下仍可能带来性能损耗。架构上，逻辑跨平台，UI 桥接原生组件。
- ** 自渲染引擎时代 **：以 **Flutter** 为代表。它不走系统的原生控件，而是自带绘图引擎（Skia），从“像素”开始绘制，性能表现极高且 UI 一致性强，但包体积较大。架构上，UI 与逻辑全跨平台，自绘渲染引擎。
- ** 原生融合 (Kotlin) 时代 **：**Kotlin** 正在开辟的新方向。它既不像 React Native 那样走桥接，也不像 Flutter 那样重写 UI，而是通过 Kotlin Multiplatform (KMP) 让一套逻辑代码（网络、数据库等）跑在不同平台，**UI 层仍然是原生的 **（iOS 用 SwiftUI/UIKit，Android 用 Jetpack Compose）。架构上，逻辑跨平台，UI 原生。

整体评价：

- WebView 容器时代基本已死，性能太差，毫无竞争力。
- KMP 和 Hybrid 方案思路非常像，但优势是消除了 JS 桥接带来的性能损耗，相对 Hybrid 方式提升了一定的性能。
- Flutter 自渲染引擎的方案在跨平台 UI 一致性这方面可谓是一枝独秀。

hls.js 和 dash.js 是绕不开的两座大山，分别实现了当今最主流的两种自适应流媒体协议：HLS 和 MPEG-DASH。

浏览器原生 `<video>` 只支持静态文件（如 MP4）和简单的 HLS（Safari 原生支持）。要实现自定义的流式加载、自适应码率、分段缓冲、低延迟直播，就必须通过 **MSE API** 动态地向播放器喂数据。

协议解析与流管理

- **Manifest 解析**：读取 `.m3u8` 或 `.mpd` 文件，获取分片列表、码率、加密信息等。
- **分片下载与调度**：根据网络状况（ABR 算法）请求合适码率的分片。
- **数据封装解析**：解析 TS/MP4 分片，提取音频、视频样本，封装成 MSE 能吃的 `SourceBuffer`。
- **直播滑动窗口**：动态移除旧数据，保证内存不过度膨胀。

例如，尝试基于 hls.js 的简化版：解析 m3u8 → 下载 TS 分片 → 用 `mux.js` 或原生 `MediaSource` 转封装为 fMP4 → 喂给 video 标签。
