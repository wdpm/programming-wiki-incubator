# 微服务和RPC框架区别

## 角色不同

- 微服务是一个完整的架构模式，解决的是单体大应用的拆分问题。
- RPC 框架是微服务架构中的通信工具，用于实现服务间的远程调用（如 Dubbo、gRPC），它不关心服务如何拆分。

## 关注点不同

微服务需要解决的问题：

- 服务如何拆分（领域驱动设计 DDD）。
- 如何独立部署和扩展（K8s + Docker）。
- 如何保证数据一致性（Saga、分布式事务）。
- 如何治理服务（服务发现、熔断、限流、跟踪、监控）。

RPC 框架需要解决的问题：

- 如何序列化数据（JSON、Protobuf）。
- 如何传输数据（TCP/HTTP/HTTP2）。
- 如何负载均衡（随机、轮询、一致性哈希）。

## 依赖关系

微服务架构依赖 RPC 框架，RPC 框架不依赖微服务。

## 常见误区

- 误区1：认为 RPC 框架 = 微服务。
  纠正：RPC 是微服务的通信手段之一，微服务还可以用 REST、消息队列等通信。

- 误区2：认为用了 RPC 就必须用微服务。
  纠正：RPC 也可用于非微服务架构（如分布式计算、多进程协作）。
